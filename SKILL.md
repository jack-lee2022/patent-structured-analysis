# Patent Structured Analysis Skill

## Purpose

Decompose any patent (PDF, text, or JSON) into a structured report using a rigorous 4-step methodology. This is the same workflow used in the patent-agent backend for competitive intelligence and FTO studies.

## When to Use

- Freedom-to-Operate (FTO) analysis before product launch
- Competitive patent teardown / reverse engineering
- Design-around / invalidity research
- Patent landscape mapping for investor due diligence
- Technical due diligence for M&A or licensing

## 4-Step Analysis Workflow

### Step 1 — Independent Claims Analysis

**Goal**: Extract the technical problem, solution, and essential elements from the independent claim(s).

**Required fields** (always present in output):

```json
{
  "step1_independent_claims_analysis": {
    "technical_problem": "What real-world problem does the patent solve?",
    "technical_solution": "How does the invention solve it? One sentence.",
    "technical_elements": ["element A", "element B", "element C"],
    "key_terms": ["term 1", "term 2"],
    "function_and_purpose": "Why was this invented?",
    "scope_and_limitations": "What does the claim explicitly exclude or constrain?"
  }
}
```

**Rules**:
- Read the independent claim(s) only (ignore dependent claims for this step).
- If multiple independent claims exist, analyze each and merge into a single set of `technical_elements`.
- `technical_elements` must be nouns or noun phrases (the "what" of the invention), not actions.
- `key_terms` are the words that define the boundary of the patent — terms a competitor would use to search for prior art.
- `scope_and_limitations` captures any geographic, temporal, or functional limits mentioned in the claim.

**Implementation note**: When parsing patent text programmatically, claims often start with "What is claimed is:" followed by numbered paragraphs. The `scripts/analyze_patent.py` extractor handles this preamble automatically. See `references/example_analysis.json` for a real parsed output.

### Step 2 — Necessary Components Mapping

**Goal**: Map each technical element to a concrete component, classify as optional or mandatory, and describe its function.

**Required fields**:

```json
{
  "step2_necessary_components": [
    {
      "component_name": "fluid-filled bulb",
      "is_optional": false,
      "function": "to sense pressure generated by the tongue and hand"
    }
  ]
}
```

**Rules**:
- Every element from `technical_elements` must appear here.
- `is_optional`: `false` if removing the component breaks the invention; `true` if the claim allows alternatives or omitting it.
- For standard "comprising" claims, listed elements are mandatory (the claim is open-ended but the listed components are required). Only mark `true` if the claim explicitly says "optionally" or "may".
- `function` should be a verb phrase starting with "to ..." (infinitive form).
- If a component is implicit in the claim but not named (e.g., "a sensor" implies "signal processing unit"), add it and mark `is_optional` based on claim language.

**Implementation note**: When searching the description for component function, use the first 3-4 words of the component name as the search key (after stripping trailing "wherein" / "configured to" / "for" clauses). This avoids missing matches due to long component names.

### Step 3 — Diagram Correspondence

**Goal**: For each necessary component, identify which figure(s) illustrate it and describe what the figure shows.

**Required fields**:

```json
{
  "step3_corresponding_diagrams": [
    {
      "component_name": "fluid-filled bulb",
      "diagram_reference": "Figure 1",
      "diagram_description": "a cross-sectional view of the fluid-filled bulb"
    }
  ]
}
```

**Rules**:
- `diagram_reference` must be the exact label used in the patent (e.g., "FIG. 1", "Figure 2", "Fig. 3A").
- If a component is not shown in any figure, write `"diagram_reference": "Not shown"` and explain in `diagram_description`.
- Multiple components can reference the same figure — this is expected.
- `diagram_description` should be a noun phrase (no verbs) describing the visual content.

**Implementation note**: US patents use "FIG. X" format. The script searches for figure references in the same sentence as the component mention.

### Step 4 — Key Diagram Identification

**Goal**: Select the single most representative figure (the "hero figure") and explain why it captures the essence of the invention.

**Required fields**:

```json
{
  "step4_key_diagrams": {
    "primary_diagram": "Figure 1",
    "reasoning": "The primary diagram shows the fluid-filled bulb, which is the core component of the invention",
    "completeness_score": 8,
    "alternative_diagrams": ["Figure 2", "Figure 3"]
  }
}
```

**Rules**:
- `primary_diagram` must be a figure referenced in Step 3.
- `reasoning` must justify why this figure is the "hero figure" — it should reference the core innovation or the element that distinguishes the invention from prior art.
- `completeness_score`: integer 1–10. 10 = the figure alone is sufficient to understand the invention without reading the text. 5 = the figure shows the invention but requires text to understand novelty. 1 = the figure is decorative or partial.
- `alternative_diagrams`: list of other figures that, if the primary figure were missing, could serve as the hero figure. Usually 1–3 items.

## Overall Summary

After the 4 steps, always add a bilingual summary:

```json
{
  "overall_summary": "本专利提出了一种便携式压力检测设备，用于测量人体某些部位（如口腔和手）所产生的压力。该设备采用液体填充的球体来感知压力，并具有转换器、压力比较电路和峰值压力检测电路等组件。该设备可用于评估和诊断各种语音障碍和病理条件。"
}
```

**Rules**:
- Summary must be in the original language of the patent (Chinese for CN/TW patents, Japanese for JP, English for US/EP).
- Length: 80–150 characters in Chinese/Japanese, 40–80 words in English.
- Must include: the invention category, the core mechanism, and the primary application.

## Input Handling

### From PDF

If the user provides a PDF path:
1. Use `pdfplumber` or `PyMuPDF` to extract text.
2. Prefer the "claims" section, then "description", then "abstract".
3. If the PDF is scanned (no extractable text), inform the user that OCR is needed.

### From Text/JSON

If the user provides raw text or a JSON object:
1. Use it directly if it contains claims and description.
2. If only abstract is provided, note that the analysis will be limited (Step 2 and 3 may be incomplete).

### From Patent Database

If the patent is already in the patent-agent database (`patents.db`):
1. Query the database for `patent_id`, `claims`, `description`, `abstract`.
2. Use the `claims` field for Step 1 and `description` for Step 2–3.

**Implementation note**: The bundled `scripts/analyze_patent.py` handles all three input sources. See `references/workflow.md` for CLI usage.

## Output Format

Always output a single JSON object with the exact structure shown above. No markdown wrappers around the JSON. The top-level keys must be:

```json
{
  "patent_id": "US1234567B2",
  "title": "Patent Title",
  "step1_independent_claims_analysis": { ... },
  "step2_necessary_components": [ ... ],
  "step3_corresponding_diagrams": [ ... ],
  "step4_key_diagrams": { ... },
  "overall_summary": "..."
}
```

## Error Handling

- If claims are missing or unreadable: return JSON with `"error": "Claims not found or unreadable"` and fill Steps 2–4 with `"N/A"`.
- If figures are missing: return `"step4_key_diagrams.primary_diagram": "None"` and `"reasoning": "No figures available in source"`.
- If the patent is not a utility patent (e.g., design patent): return `"error": "Design patents are not supported by this analysis workflow"`.
